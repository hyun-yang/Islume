Decision for Islume

Railway was chosen. The reasoning: Islume's agent matching creates bursty workloads (periods of high activity followed by idle time), which fits Railway's usage-based pricing perfectly. Scale-to-zero means unused LLM workers cost nothing. The visual canvas is particularly valuable for a multi-service system like Islume (5 services + databases). The 5-minute HTTP timeout is not a concern because agent conversations are processed asynchronously via Redis Streams, so HTTP requests themselves remain short — only the background workers run for extended periods.

2. System Overview

Islume is decomposed into five independently scalable services connected via Redis (Streams + GEO) and Postgres (persistent state). The architecture follows two principles: Redis owns "what's happening now" (positions, session state, task queue, live event streams), and Postgres owns "what already happened" (completed conversations, user accounts, agent definitions).

3. Agent Orchestrator

The Orchestrator is the brain of Islume — it receives match events, creates conversation sessions, loads persona prompts, and manages the turn-by-turn dialogue loop between two matched agents.

Fig 2 — Orchestrator pipeline: match → session → turns → LLM

Component Responsibilities

Session Manager creates a session record binding two agents, their match context, a TTL (e.g. 300s), and initial turn counter. Session state lives in Redis Hash for real-time access; persistent record in Postgres.

Persona Loader fetches both agents' persona templates from Postgres and injects the match context dynamically. This makes the same persona behave differently depending on who they're matched with.

Turn Scheduler implements a simple state machine: even turns → Agent A, odd turns → Agent B. Each turn enqueues an LLM task to Redis Streams, and the result triggers the next turn until TTL expires or a user disconnects.

4. Map + Matching Service

The Map + Matching service handles real-time island positioning on the shared map and pairs nearby islands with similar agent personas. It runs a dual-filter pipeline: geographic proximity via Redis GEOSEARCH, then persona similarity scoring.

Fig 4 — Matching pipeline (4-stage filter)

4.1 Matching Illustration

The following diagram shows a concrete matching scenario. Alice (Jazz Lover) is at the center. Three other users are in the area, and the system computes similarity scores against each.

Fig 5 — Dual-filter matching: proximity + persona similarity

Matching algorithm steps:

1. Proximity filter uses GEOSEARCH to return all islands within the radius (default 500m). User E is excluded because out of range.

2. Bulk agent load uses a single SQL query to fetch active agents for all nearby candidates (avoids N+1).

3. Similarity scoring computes Jaccard similarity on tag sets. For Alice (jazz) vs Bob (vinyl), tag overlap of 3 (music, vinyl, analog) over union of 7 yields ~0.43 — above the 0.3 threshold. User C (gamer) has no tag overlap and is dropped.

4. Match resolver handles three critical edge cases: duplicate prevention via Redis SETNX locks, cooldown tracking via sorted sets (blocking repeat matches within 30 minutes), and session limit enforcement (one active session per agent).

6. LLM Worker Pool

The LLM Worker Pool is the most cost-intensive component. It consumes tasks from Redis Streams, calls LLM APIs to generate persona replies, and writes to both the database and the session event stream.

Fig 8 — LLM worker pool internal structure

6.1 Worker Internal Pipeline

A single turn task flows through the worker in seven steps. The colors encode which subsystem is touched: coral for Python-only work, teal for database and stream writes, purple for loop control.

Fig 9 — Worker internal pipeline: parse → load → LLM → persist → publish → loop

Perspective swap. Step 3 is the most subtle part. Anthropic's API expects messages in a specific format where assistant is the model's own past turns. Since two different personas share one conversation, the worker swaps perspectives when loading history: from speaker A's view, A's turns are assistant and B's turns are user. From speaker B's view, the labels are mirrored.

Publish order matters. Step 5 (DB commit) must come before Step 6 (stream publish). If the order were reversed, a gateway client could see a turn event referring to a turn that isn't yet persisted — and a subsequent page reload would show that turn missing. The database is the source of truth; the stream is just a notification channel.

Self-perpetuating queue. Step 7 is the key pattern. The worker enqueues the next speaker's turn into the same stream:llm_tasks, where another worker (or itself) will pick it up. This eliminates the need for any external coordinator. The orchestrator only ever enqueues the first task of a session.

6.2 Context Builder Strategy

As conversations progress, sending the full history to the LLM would explode token costs. The sliding window strategy keeps the last 6 turns as full text and compresses earlier turns into a summary (generated by a cheaper model like Haiku). This achieves ~90% token savings on older context.

6.3 LLM Router Design

The LLM Router is a small dispatcher that decides which model to use for each task type. In the MVP there is no router — all calls go to Sonnet 4.5. But as the system grows to include summarization, consistency checks, and quality scoring, a router becomes essential for controlling cost without hardcoding model names across the codebase.

Fig 11 — LLM router decision flow

Task Type Policies

TaskPrimaryFallbackWhy
Persona replySonnet 4.5Haiku 4.5Quality first; fall back if Sonnet is rate-limited
History summaryHaiku 4.5NoneInternal use only; Haiku is cheap enough
Match contextHaiku 4.5NoneOne-sentence output; cacheable
Consistency checkHaiku 4.5NoneClassification task, not generation
Safety checkHaiku 4.5NoneClassification task, low latency needed
Quality scoreOpus 4.6Sonnet 4.5Batch analytics, accuracy matters most

The router's job is to centralize these decisions. Calling code says "this is a persona reply task"; the router picks the model. This makes it easy to change model assignments, A/B test alternatives, and enforce per-session budget caps without touching the worker logic.

7.1 Consumer Group Mental Model

The most important thing to understand about Redis Streams consumer groups: the stream and the consumer group are separate concepts. The stream is just an append-only log. The consumer group is a read-state manager layered on top, tracking which consumer got which message and whether it has been acknowledged.

Why Streams Instead of Pub/Sub?

An early version of Islume used Redis Pub/Sub to deliver turn events from worker to gateway. It failed intermittently: messages were silently lost during the window between pubsub.subscribe() and the first pubsub.listen() iteration. Pub/Sub is fire-and-forget — a subscriber that connects after a message was published will never see it. Streams solve this structurally: messages are durable, and clients fetch from a known position using XREAD. Replay and live use the same mechanism. See the JOURNAL for the full debugging journey.

8. Database Schema

Postgres handles persistent data — user accounts, agent definitions, match history, and conversation logs. Redis handles all real-time state. The boundary is clear: Redis for "what's happening now," Postgres for "what already happened."

Fig 15 — Postgres schema: users, agents, sessions, conversation turns

Index Strategy

PriorityIndexPurpose

hotuser_agents (user_id) partialActive agent per user

warmmatch_sessions (island_a) WHERE status='active'Active sessions per island

Partitioning note: The conversation_turns table grows fastest (~30,000 rows/month at moderate usage). Monthly partitioning on created_at keeps analytics queries fast and simplifies data retention. Not yet implemented in MVP.

9. Local Development Environment

Islume's local development runs directly on Ubuntu 24.04 host: infrastructure services (Postgres, Redis) as Docker Compose containers, application code (Python services) natively via uv run. No VMware. This keeps the feedback loop fast while still isolating stateful services.

Fig 16 — Local development stack on Ubuntu host

See the project's README for full setup instructions. Essential tools: Docker + Docker Compose, uv (Python package manager), and an Anthropic API key. All infrastructure starts with a single docker compose up -d command.
