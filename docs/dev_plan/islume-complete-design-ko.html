Islume의 선택

Railway를 선택했어요. 이유는 이래요. Islume의 에이전트 매칭은 burst 워크로드를 만들어요. 활발한 시간과 idle한 시간이 반복되는 패턴이죠. 이게 Railway의 사용량 기반 가격에 완벽히 맞아요. Scale-to-zero 덕분에 사용하지 않는 LLM 워커는 비용이 0이에요. Islume처럼 멀티 서비스 시스템(5개 서비스 + 데이터베이스)에서는 비주얼 캔버스가 특히 유용해요. 5분 HTTP 타임아웃 제한은 문제가 되지 않아요. 에이전트 대화는 Redis Streams를 통해 비동기로 처리되기 때문에, HTTP 요청 자체는 짧게 유지되거든요. 오직 백그라운드 워커만 길게 실행돼요.

2. 시스템 개요

Islume은 다섯 개의 독립적으로 확장 가능한 서비스로 분해되어 있고, Redis(Streams + GEO)와 Postgres(영속 상태)로 연결돼요. 아키텍처는 두 가지 원칙을 따라요. Redis는 "지금 일어나고 있는 일"을 소유해요. 위치, 세션 상태, 작업 큐, 라이브 이벤트 스트림이 여기에 해당해요. Postgres는 "이미 일어난 일"을 소유해요. 완료된 대화, 사용자 계정, 에이전트 정의가 여기에 들어가요.

3. 에이전트 오케스트레이터

오케스트레이터는 Islume의 두뇌예요. 매칭 이벤트를 받아 대화 세션을 생성하고, 페르소나 프롬프트를 로드하며, 두 매칭된 에이전트 사이의 턴별 대화 루프를 관리해요.

그림 2 — 오케스트레이터 파이프라인: 매칭 → 세션 → 턴 → LLM

컴포넌트 책임

세션 매니저는 두 에이전트, 매칭 컨텍스트, TTL(예: 300초), 초기 턴 카운터를 묶는 세션 레코드를 생성해요. 세션 상태는 실시간 접근을 위해 Redis Hash에 저장되고, 영속 레코드는 Postgres에 들어가요.

페르소나 로더는 양쪽 에이전트의 페르소나 템플릿을 Postgres에서 가져와 매칭 컨텍스트를 동적으로 주입해요. 이 덕분에 같은 페르소나가 누구와 매칭됐는지에 따라 다르게 행동할 수 있어요.

턴 스케줄러는 단순한 상태 머신을 구현해요. 짝수 턴은 에이전트 A, 홀수 턴은 에이전트 B로 가요. 각 턴마다 Redis Streams에 LLM 작업을 등록하고, 결과는 다음 턴을 트리거해요. TTL 만료 또는 사용자 연결 해제까지 계속돼요.

4. 맵 + 매칭 서비스

맵 + 매칭 서비스는 공유 맵 위의 실시간 섬 위치 관리와, 유사한 에이전트 페르소나를 가진 가까운 섬들을 페어링하는 작업을 담당해요. 이중 필터 파이프라인을 실행해요. Redis GEOSEARCH를 통한 지리적 근접 필터와, 그 다음 페르소나 유사도 점수 계산이에요.

그림 4 — 매칭 파이프라인 (4단계 필터)

4.1 매칭 일러스트레이션

다음 다이어그램은 구체적인 매칭 시나리오를 보여줘요. Alice(재즈 애호가)가 중앙에 있고, 주변에 다른 3명의 사용자가 있어요. 시스템은 각각에 대해 유사도 점수를 계산해요.

그림 5 — 이중 필터 매칭: 근접도 + 페르소나 유사도

매칭 알고리즘 단계:

1. 근접 필터는 GEOSEARCH를 사용해 반경(기본 500m) 내의 모든 섬을 반환해요. 사용자 E는 범위 밖이라 제외돼요.

2. 벌크 에이전트 로드는 단일 SQL 쿼리로 모든 인접 후보의 활성 에이전트를 가져와요(N+1 회피).

3. 유사도 점수 계산은 태그 집합에 대해 Jaccard 유사도를 구해요. Alice(재즈)와 Bob(바이닐)의 경우, 태그 공통 3개(음악, 바이닐, 아날로그)에 합집합 7개로 ~0.43이에요. 이는 0.3 임계값을 넘어요. 사용자 C(게이머)는 태그 공통이 없어 제외돼요.

4. 매칭 리졸버는 세 가지 결정적 엣지 케이스를 처리해요. Redis SETNX 잠금을 통한 중복 방지, 정렬된 셋을 통한 쿨다운 추적(30분 내 반복 매칭 차단), 그리고 세션 한도 강제(에이전트당 활성 세션 1개)예요.

5. WebSocket 게이트웨이

WebSocket 게이트웨이는 실시간 통신 허브예요. 맵 위의 섬 위치를 동기화하고, 매칭/대화 이벤트를 즉시 전달해요. FastAPI WebSocket 핸들러와 redis.asyncio로 구축했어요.

6. LLM 워커 풀

LLM 워커 풀은 가장 비용이 큰 컴포넌트예요. Redis Streams에서 작업을 가져와 LLM API를 호출해 페르소나 응답을 생성하고, 데이터베이스와 세션 이벤트 스트림 양쪽에 저장해요.

그림 8 — LLM 워커 풀 내부 구조

6.1 워커 내부 파이프라인

하나의 턴 작업이 워커 안에서 7단계로 흘러가요. 색상은 어떤 서브시스템이 관여하는지 인코딩해요. 코랄은 Python-only 작업, 틸은 데이터베이스와 스트림 쓰기, 퍼플은 루프 제어예요.

그림 9 — 워커 내부 파이프라인: 파싱 → 로드 → LLM → 영속화 → 발행 → 루프

관점 교체. 3단계가 가장 미묘한 부분이에요. Anthropic API는 assistant가 모델 자신의 과거 턴인 특정 포맷을 기대해요. 두 개의 페르소나가 한 대화를 공유하므로, 워커는 히스토리 로드 시 관점을 교체해야 해요. 스피커 A 관점에서 A의 턴은 assistant이고 B의 턴은 user예요. 스피커 B 관점에서는 라벨이 반대예요.

발행 순서가 중요해요. 5단계(DB 커밋)는 6단계(스트림 발행)보다 먼저 와야 해요. 순서가 반대였다면, 게이트웨이 클라이언트가 아직 영속화되지 않은 턴을 참조하는 이벤트를 받을 수 있어요. 그 후 페이지 새로고침을 하면 그 턴이 사라져 보일 수 있죠. 데이터베이스가 진실의 원천이고, 스트림은 알림 채널일 뿐이에요.

자체 영속화 큐. 7단계가 핵심 패턴이에요. 워커는 다음 스피커의 턴을 같은 stream:llm_tasks에 등록해요. 그러면 다른 워커(또는 자기 자신)가 이를 가져가요. 이 덕분에 외부 코디네이터가 필요 없어요. 오케스트레이터는 오직 세션의 첫 번째 작업만 등록해요.

6.2 컨텍스트 빌더 전략

6.3 LLM 라우터 설계

LLM 라우터는 각 작업 타입에 어떤 모델을 사용할지 결정하는 작은 디스패처예요. MVP에는 라우터가 없어요. 모든 호출이 Sonnet 4.5로 가요. 하지만 시스템이 요약, 일관성 체크, 품질 점수 같은 기능을 포함하도록 성장하면, 코드베이스 전체에 모델 이름을 하드코딩하지 않으면서 비용을 통제하기 위해 라우터가 필수가 돼요.

라우터의 역할은 이런 결정을 중앙화하는 거예요. 호출하는 쪽이 "이건 페르소나 응답 작업이다"라고 말하면, 라우터가 모델을 선택해요. 그러면 모델 할당 변경, A/B 테스트, 세션별 예산 한도 강제 같은 일이 워커 로직을 건드리지 않고도 쉬워져요.

7. Redis 메시지 큐 패턴

Redis는 Islume에서 세 가지 역할을 동시에 수행해요. Streams는 영속적인 비동기 작업 큐잉, 세션 스트림은 세션별 이벤트 로그, GEO/Hash/Set은 휘발성 상태 저장이에요. 초기에는 단일 Redis 인스턴스가 세 역할을 모두 처리해요.

왜 Pub/Sub 대신 Streams를 쓰는가

Islume 초기 버전은 워커에서 게이트웨이로 턴 이벤트를 전달하는 데 Redis Pub/Sub을 사용했어요. 이게 산발적으로 실패했어요. pubsub.subscribe()와 첫 번째 pubsub.listen() 반복 사이의 윈도우 동안 메시지가 조용히 손실됐어요. Pub/Sub은 fire-and-forget이에요. 메시지가 발행된 후에 연결된 구독자는 그 메시지를 절대 볼 수 없어요. Streams는 이걸 구조적으로 해결해요. 메시지가 영속적이고, 클라이언트는 XREAD로 알려진 위치에서 가져와요. Replay와 live가 같은 메커니즘을 써요. 전체 디버깅 여정은 JOURNAL을 참고하세요.

8. 데이터베이스 스키마

Postgres는 영속 데이터를 처리해요. 사용자 계정, 에이전트 정의, 매칭 히스토리, 대화 로그가 들어가요. Redis는 모든 실시간 상태를 처리해요. 경계가 명확해요. Redis는 "지금 일어나고 있는 일"용, Postgres는 "이미 일어난 일"용이에요.

9. 로컬 개발 환경

Islume의 로컬 개발은 Ubuntu 24.04 호스트에서 직접 실행돼요. 인프라 서비스(Postgres, Redis)는 Docker Compose 컨테이너로, 애플리케이션 코드(Python 서비스)는 uv run으로 네이티브 실행해요. VMware는 사용하지 않아요. 이 구성이 피드백 루프를 빠르게 유지하면서도 stateful 서비스를 격리해요.
